# Phase Launch189 - Crowd Pop Edit Guidance

## Goal

Improve highlight recall for loud crowd and court-audio moments without violating the cloud-first editing boundary.

## Architecture

- iOS does not analyze, render, or classify video/audio locally.
- iOS sends compact edit intent to the cloud AI Edit backend.
- Backend candidate generation can surface loud crowd/court audio pops as recall anchors.
- GPT/edit agents may use those audio cues only after validating sampled keyframes show basketball action and a visible outcome.
- Deterministic validators and renderers still own timestamps, EditPlan safety, and final rendering.

## Change

- Added crowd/audio-pop guidance to the default AI Edit prompt generated by `CloudEditUserPromptBuilder`.
- Added the same cue to the visible AI Edit focus summary so testers understand why loud crowd moments can be reviewed.
- Kept guardrails explicit: crowd/audio pops are cues only, and the cloud editor must verify visible play and outcome.

## Backend Context

The backend already supports this path:

- `ios/backend/app/pipeline.py` detects loud local pops and sustained crowd-swell onsets.
- Candidate windows use the `Crowd Reaction` label as recall candidates for GPT review.
- `ios/backend/app/editing.py` and `services/editing/editing_app/gpt_reranker.py` instruct GPT that crowd/audio pops are recall hints only.
- Existing tests reject crowd-reaction outcome claims when sampled frames do not support the event.

## Launch Recommendation

Keep the backend policy wording strict: audio reactions should improve candidate recall, not automatically create final highlights. For uncertain but promising crowd-pop clips, keep them reviewable instead of silently discarding them.
