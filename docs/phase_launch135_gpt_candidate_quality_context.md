# Phase Launch135 GPT Candidate Quality Context

## Goal

Improve GPT-led highlight clipping quality by giving the cloud AI edit agent clearer compact candidate signals before it selects, rejects, captions, orders, or patches an `EditPlan`.

## Change

The backend `build_agent_editing_context(...)` payload now includes:

- `candidateQualitySummary`
  - candidate count
  - render-ready count
  - manual team-review count
  - quality-eligible count
  - shot-like count
  - defensive-event count
- Per-candidate `planningScore`
- Per-candidate `renderEligibility`
  - `render_ready`
  - `render_ready_after_user_keep`
  - `manual_team_review_required`
  - `quality_review_required`
  - `opponent_filtered`
  - `user_discarded`
- Per-candidate `candidateQuality`
  - `qualityEligible`
  - `contextQualityScore`
  - `outcomeReliabilityScore`
  - `shotLike`
  - `defensiveEventLike`
  - `defensiveFamily`
  - `genericFiller`

## Why This Helps Accuracy

GPT already receives candidate clips, sampled-frame evidence, team attribution, and native shot signals. This change makes the editing prompt easier for the model to judge:

- which clips are strong enough to render
- which clips have complete setup/outcome context
- which clips are defensive highlights such as blocks or steals
- which selected-team clips are review-only because team attribution is uncertain
- which generic filler clips should be rejected

That should improve semantic editing without asking GPT to replace CV, FFmpeg, deterministic timestamps, validators, or render logic.

## Architecture Check

- Cloud backend only.
- Existing candidate clips only.
- No full video sent to GPT.
- No FFmpeg commands generated by GPT.
- No iOS analysis/rendering/composition changes.
- Existing validators and deterministic renderer remain the source of truth.
